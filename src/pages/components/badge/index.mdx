## Overview


### When to Use Badges
- **Status indication**: Online/offline, active/inactive, approval states
- **Notifications**: Unread counts, new items, urgent alerts
- **Categorization**: Tags, labels, content classification
- **Metadata**: Version numbers, types, feature flags
- **Highlighting**: Important or featured content that needs attention

### When NOT to Use Badges
- **Primary actions**: Use Button component for main interactions
- **Navigation**: Use Link or navigation components for page transitions
- **Large text content**: Use appropriate text components for substantial information
- **Complex information**: Use Cards or other layout components for detailed data

## Design Principles


Understanding these core principles will help you make better design decisions and avoid interface patterns that create visual noise or confusion.

### Semantic Communication Principle
**Visual design should match functional meaning.** Badge colors and styles should intuitively communicate the type of information being presented. Misleading visual cues break user expectations and reduce system trust.

### Visual Hierarchy Principle  
**Badges should support, not compete with primary content.** They provide supplementary information without overwhelming the main interface. Overusing badges creates visual clutter and reduces their effectiveness.

### Cognitive Load Principle
**Keep badge content concise and scannable.** Users should be able to quickly understand badge meaning without stopping their workflow. Complex or lengthy badge content defeats their purpose as quick indicators.

### Consistency Principle
**Similar information should use similar badge treatments across your product.** Consistent badge usage creates predictable patterns that users can learn and rely on throughout the interface.

## Best Practices

### Common Usage Patterns

```html
<!-- Status indicators -->
<elvt-badge variant="success">Active</elvt-badge>
<elvt-badge variant="warning">Pending</elvt-badge>
<elvt-badge variant="error">Failed</elvt-badge>

<!-- Notification counts -->
<elvt-badge variant="error" count="3">Alerts</elvt-badge>
<elvt-badge count="127">Unread</elvt-badge>

<!-- Category tags -->
<elvt-badge dismissible>JavaScript</elvt-badge>
<elvt-badge dismissible>React</elvt-badge>

<!-- Metadata -->
<elvt-badge outline>v2.1.0</elvt-badge>
<elvt-badge>Beta</elvt-badge>

<!-- Dot indicators -->
<elvt-badge dot variant="success"></elvt-badge>
```

### Key Properties

The most commonly used properties for design and development:

<PageTable>

| Property | Options | Description |
|----------|---------|-------------|
| `variant` | `default`, `success`, `warning`, `error`, `info` | Visual style that communicates meaning |
| `size` | `small`, `medium` (default), `large` | Physical size of the badge |  
| `count` | Number | Displays numeric value with proper formatting |
| `dismissible` | `true`, `false` (default) | Whether the badge can be removed |
| `outline` | `true`, `false` (default) | Use outline style instead of filled |
| `dot` | `true`, `false` (default) | Show as minimal dot indicator |

</PageTable>

*For complete API reference including all properties, events, and methods, see the [Storybook documentation](https://elevate-core-ui.inform-cloud.io/?path=/docs/components-badge--docs).*

## Accessibility


### Essential Accessibility Practices

**Count badges need descriptive context:**
```html
<!-- Good: Screen readers understand what the count represents -->
<elvt-badge count="5" aria-label="5 unread messages">Messages</elvt-badge>

<!-- Bad: Screen readers can't identify what the number means -->
<elvt-badge count="5"></elvt-badge>
```

**Status badges should include text, not just color:**
- ✅ "Active", "Pending", "Failed" with semantic colors
- ⛔️ Color-only indicators without text labels

**Ensure sufficient contrast:**
- Badge text must meet 4.5:1 contrast ratio minimum
- All badge variants are designed to meet WCAG requirements

**Interactive badges need proper focus:**
- Dismissible badges have clear focus indicators
- Space and Enter activate dismissible badges
- Tab navigation works without additional code

**Don't use badges for primary actions:**
- Badges are informational, not interactive elements
- Use Button component for actions users need to take
