## Overview

*Purpose: Establish context and help designers/developers understand when and why to use this component.*

### When to Use Buttons
- **Primary actions**: Sign up, submit, save, confirm, purchase
- **Secondary actions**: Cancel, back, learn more, edit  
- **Navigation**: Next step, continue, proceed to checkout
- **Destructive actions**: Delete, remove, reset, archive

### When NOT to Use Buttons
- **Links**: Use Link component for navigation between pages or external sites
- **Toggles**: Use Switch component for on/off states or settings
- **Selection**: Use Checkbox or Radio for multiple choice selections
- **Filters**: Use FilterButton or Chip components for content filtering

## Design Principles

*Purpose: Understand the foundational UI principles that guide effective button design and prevent common mistakes.*

Understanding these core principles will help you make better design decisions and avoid interface patterns that confuse or frustrate users.

### Visual Hierarchy Principle
**Clear hierarchy guides user attention and reduces cognitive load.** Users should immediately understand which action is most important on any given screen. Competing visual weights create confusion and decision paralysis.

### Semantic Communication Principle  
**Visual design should match functional meaning.** Colors, shapes, and styling should intuitively communicate what type of action the button performs. Misleading visual cues break user expectations and decrease trust.

### Cognitive Load Principle
**Reduce the mental effort required to use your interface.** Button labels should be descriptive and action-oriented. Generic or vague labels force users to guess what will happen, increasing friction and mistakes.

### Consistency Principle
**Similar actions should look and behave similarly across your product.** Inconsistent button styling creates a fragmented experience and forces users to relearn patterns throughout your interface.

## Implementation

*Purpose: Provide developers with practical implementation guidance and common usage patterns, not exhaustive technical reference (see Storybook for complete API docs).*

### Getting Started

```bash
npm install @inform-elevate/elevate-core-ui
```

```javascript
// Import all components
import '@inform-elevate/elevate-core-ui';

// Or import specific components  
import '@inform-elevate/elevate-core-ui/dist/components/button/button.js';
```

### Common Usage Patterns

```html
<!-- Primary action with secondary support -->
<elevate-button tone="primary">Save Changes</elevate-button>
<elevate-button>Cancel</elevate-button>

<!-- Action with prefix icon -->
<elevate-button tone="primary">
  <elevate-icon slot="prefix" icon="mdi:plus"></elevate-icon>
  Add Item
</elevate-button>

<!-- Navigation with suffix icon -->
<elevate-button>
  Continue
  <elevate-icon slot="suffix" icon="mdi:chevron-right"></elevate-icon>
</elevate-button>

<!-- Destructive action -->
<elevate-button tone="danger">Delete Account</elevate-button>

<!-- Link button -->
<elevate-button href="/dashboard">Go to Dashboard</elevate-button>

<!-- Form submission -->
<elevate-button type="submit" tone="primary">Submit Form</elevate-button>
```

### Key Properties

The most commonly used properties for design and development:

<PageTable>

| Property | Options | Description |
|----------|---------|-------------|
| `tone` | `neutral` (default), `primary`, `success`, `warning`, `danger`, `emphasized`, `subtle` | Visual style that communicates action type |
| `size` | `small`, `medium` (default), `large` | Physical size of the button |  
| `shape` | `box` (default), `pill` | Corner radius style |
| `disabled` | `true`, `false` (default) | Whether the button can be interacted with |
| `href` | URL string | Makes button behave as a link |
| `type` | `button` (default), `submit`, `reset` | Form behavior when used in forms |

</PageTable>

**Icon Slots:** Use `slot="prefix"` for icons before text, `slot="suffix"` for icons after text.

*For complete API reference including all properties, events, and methods, see the [Storybook documentation](storybook-url).*

## Accessibility

*Purpose: Ensure your button implementations are usable by all users, including those with disabilities.*

### Essential Accessibility Practices

**Icon-only buttons require labels:**
```html
<!-- Good: Screen readers can announce the action -->
<elevate-button aria-label="Delete item" title="Delete item">
  <elevate-icon icon="mdi:delete"></elevate-icon>
</elevate-button>

<!-- Bad: Screen readers can't identify the action -->
<elevate-button>
  <elevate-icon icon="mdi:delete"></elevate-icon>
</elevate-button>
```

**Use clear, descriptive labels:**
- ✅ "Delete 3 selected items" 
- ❌ "OK" or "Click here"

**Ensure sufficient contrast:**
- Button text must meet 4.5:1 contrast ratio minimum
- Focus indicators must be clearly visible

**Maintain touch target size:**
- Minimum 44px × 44px for mobile interfaces
- All button sizes meet this requirement by default

**Keyboard behavior is built-in:**
- Space and Enter activate buttons automatically
- Tab navigation works without additional code
