## Overview

*Purpose: Establish context and help designers/developers understand when and why to use this component.*

### When to Use Tooltips
- **Icon explanations**: Clarify the meaning of icon-only buttons and controls
- **Form field help**: Provide formatting examples and validation requirements
- **Feature descriptions**: Explain complex functionality without cluttering the UI
- **Additional context**: Offer supplementary information about interface elements
- **Keyboard shortcuts**: Display available shortcuts for power users

### When NOT to Use Tooltips
- **Essential information**: Critical information should not be hidden in tooltips
- **Mobile interfaces**: Touch devices don't have hover states for tooltip activation
- **Long content**: Use modals or expandable sections for substantial information
- **Repeated interactions**: Information needed frequently should be always visible

### Trigger Behaviors

**Hover Activation** shows tooltips on mouse hover with configurable delay.

**Focus Activation** displays tooltips when elements receive keyboard focus.

**Click Activation** toggles tooltips on click for touch-friendly interfaces.

**Manual Control** allows programmatic tooltip display and hiding.

## Design Principles

*Purpose: Understand the foundational UI principles that guide effective tooltip design and prevent common mistakes.*

### Discoverability Principle
**Users should be able to find helpful information when they need it.** Tooltips should be available on elements that might benefit from explanation, but not so ubiquitous that they become annoying.

### Timing Principle  
**Tooltip appearance and disappearance should feel natural and responsive.** Too-fast activation creates accidental triggers while too-slow activation feels unresponsive to user intent.

### Clarity Principle
**Tooltip content should be immediately useful and clearly written.** Vague or redundant tooltip text wastes user attention and undermines the helpfulness of the tooltip system.

### Non-Interference Principle
**Tooltips should enhance rather than interrupt user workflows.** They should provide help without blocking content or interfering with ongoing tasks.

## Best Practices

*Purpose: Apply design principles through specific, actionable guidelines that prevent common tooltip design mistakes.*

### Writing Effective Tooltip Content

**Why this matters:** Poor tooltip content provides no value and can annoy users. Good tooltips should be concise, informative, and genuinely helpful rather than stating the obvious.

### Choosing Appropriate Triggers

**Why this matters:** Wrong trigger methods can make tooltips inaccessible or annoying. Hover works well for desktop but not mobile, while click activation might interfere with primary element functions.

### Managing Tooltip Positioning

**Why this matters:** Poorly positioned tooltips can be cut off by viewport edges or cover important content. Smart positioning ensures tooltips remain readable and useful.

### Planned Usage Patterns

```html
<!-- Basic tooltip -->
<elvt-tooltip content="Save your changes">
  <elvt-button>
    <elvt-icon icon="mdi:content-save"></elvt-icon>
  </elvt-button>
</elvt-tooltip>

<!-- Rich content tooltip -->
<elvt-tooltip>
  <elvt-button slot="trigger">Help</elvt-button>
  <div slot="content">
    <h4>Keyboard Shortcuts</h4>
    <ul>
      <li><kbd>Ctrl+S</kbd> - Save</li>
      <li><kbd>Ctrl+Z</kbd> - Undo</li>
    </ul>
  </div>
</elvt-tooltip>

<!-- Form field tooltip -->
<div>
  <label for="password">Password</label>
  <elvt-tooltip content="Must be at least 8 characters with numbers and symbols">
    <input type="password" id="password" />
  </elvt-tooltip>
</div>

<!-- Positioned tooltip -->
<elvt-tooltip content="Additional options" position="bottom">
  <elvt-button>
    <elvt-icon icon="mdi:dots-vertical"></elvt-icon>
  </elvt-button>
</elvt-tooltip>

<!-- Click-activated tooltip -->
<elvt-tooltip content="Click to copy link" trigger="click">
  <elvt-button>Share Link</elvt-button>
</elvt-tooltip>
```

## Accessibility

*Purpose: Ensure your tooltip implementations are usable by all users, including those with disabilities.*

### Essential Accessibility Practices

**Keyboard accessibility:**
- Tooltips should appear on focus for keyboard users
- Escape key should dismiss active tooltips
- Tab navigation should work predictably with tooltips
- Interactive tooltip content needs proper focus management

**Screen reader support:**
- Use `aria-describedby` to associate tooltips with trigger elements
- Ensure tooltip content is announced by screen readers
- Don't use tooltips for critical information that screen readers might miss
- Provide alternative ways to access tooltip information

**Focus management:**
- Don't trap focus within tooltips unless they contain interactive elements
- Return focus appropriately when tooltips are dismissed
- Ensure tooltips don't interfere with normal tab order
- Provide clear focus indicators for tooltip triggers

**Alternative access methods:**
- Consider that mobile users can't hover to see tooltips
- Provide alternative ways to access tooltip information on touch devices
- Don't rely solely on tooltips for important help information
- Test tooltip functionality with keyboard-only navigation
