## Overview


### When to Use Icons
- **Action identification**: Buttons, menu items, and interactive elements
- **Status indication**: Success, warning, error, and information states
- **Navigation aids**: Wayfinding and section identification
- **Content categorization**: Document types, content themes, and classifications
- **Visual reinforcement**: Supporting text with recognizable symbols

### When NOT to Use Icons
- **Unclear concepts**: Avoid icons for abstract or ambiguous ideas
- **Cultural specificity**: Don't use icons that may not translate across cultures
- **Decoration only**: Icons should serve functional purposes, not just visual appeal
- **Overcrowding**: Too many icons create visual noise and reduce effectiveness

## Variations


### Icon Categories

**Action Icons** represent user interactions like save, delete, edit, and share.

**Navigation Icons** help users move through interfaces with arrows, home, and menu symbols.

**Status Icons** communicate states through checkmarks, warnings, errors, and information symbols.

**Content Icons** represent different types of content like documents, images, and media files.

**System Icons** indicate system functions like settings, search, and notifications.

### Size Variations

**Small Icons (16px)** work well in dense interfaces and inline with text.

**Medium Icons (24px)** provide standard icon size for most interface elements.

**Large Icons (32px)** create emphasis and work well for primary actions.

**Extra Large Icons (48px+)** serve as focal points and illustration elements.

### Style Treatments

**Outlined Icons** use strokes to define shapes, providing clean, minimal appearance.

**Filled Icons** use solid shapes for greater visual weight and emphasis.

**Two-Tone Icons** combine outlined and filled elements for visual hierarchy.

## Design Principles


### Recognition Principle
**Icons should be immediately recognizable and universally understood.** Users should not have to guess what icons represent. Unclear or overly stylized icons create cognitive load and slow down task completion.

### Consistency Principle  
**Similar functions should use similar icons throughout the interface.** Inconsistent icon usage confuses users and reduces the efficiency benefits that icons provide.

### Contextual Relevance Principle
**Icons should enhance rather than replace clear communication.** Icons work best when they support text or have established conventions, not when they carry full communication responsibility.

### Visual Hierarchy Principle
**Icon size and treatment should reflect importance and interaction priority.** Primary actions might use larger or more prominent icons while secondary functions use smaller, subtler treatments.

## Best Practices


### Choosing Recognizable Icons

**Why this matters:** Unfamiliar icons force users to learn new visual vocabulary, increasing cognitive load. Well-established icon conventions help users understand functionality instantly.

### Pairing Icons with Text

**Why this matters:** Icons alone can be ambiguous, especially for complex or abstract concepts. Text labels provide clarity and improve accessibility for all users.

### Maintaining Visual Consistency

**Why this matters:** Mixed icon styles create visual inconsistency that makes interfaces feel unprofessional and harder to scan. Consistent treatment improves overall design cohesion.

### Key Properties

The most commonly planned properties for design and development:

<PageTable>

| Property | Options | Description |
|----------|---------|-------------|
| `icon` | Icon name string | Icon identifier from supported icon libraries |
| `size` | Number or preset | Icon size in pixels or named size |  
| `color` | Color value or theme token | Icon color, supports theme colors |
| `interactive` | `true`, `false` (default) | Whether icon responds to hover/focus |
| `aria-label` | String | Accessibility label for screen readers |
| `title` | String | Tooltip text for icon meaning |

</PageTable>

## Accessibility


### Essential Accessibility Practices

**Provide alternative text:**
```html
<!-- Decorative icons (no screen reader announcement) -->
<elvt-icon icon="mdi:star" aria-hidden="true"></elvt-icon>

<!-- Functional icons (with meaning) -->
<elvt-icon icon="mdi:delete" aria-label="Delete item"></elvt-icon>
<elvt-icon icon="mdi:edit" title="Edit item"></elvt-icon>
```

**Don't rely solely on icons:**
- Always provide text labels or context for important actions
- Use icons to enhance, not replace, clear communication
- Consider that icon meanings may not be universal

**Interactive icon accessibility:**
- Interactive icons need proper focus indicators
- Use appropriate ARIA roles for clickable icons
- Ensure adequate touch target size (44px minimum)
- Provide keyboard interaction support

**Color and contrast:**
- Icons must meet 3:1 contrast ratio minimum
- Don't use color alone to convey meaning
- Ensure icons remain visible in high contrast mode
